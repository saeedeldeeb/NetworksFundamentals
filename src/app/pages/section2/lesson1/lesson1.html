Client-Server Architecture

The revolution that started it all — separating server and client into different locations where different pieces of code can live and execute remotely.

The Problem It Solves

  • Machines are expensive
  • Applications are complex
  • Running everything on a single mainframe is inefficient

Solution: Break the application into multiple components that can call each other.

Benefits

Centralized Servers

Servers have dedicated resources; clients use commodity hardware

Better Scaling

Many lightweight clients can call a single powerful server

Faster Startup

Clients don't carry heavy application logic; binaries are smaller

Fewer Dependencies

Server handles dependencies (DB drivers, libs); client just makes calls

Key insight: Microservices inherited this same model — breaking a monolith into smaller services that call each other.

Remote Procedure Call (RPC)

Born from client-server architecture in the 1960s-70s. The idea of making a call to a remote machine. Initially no standards — just send data across the wire. Standards evolved over time, leading to gRPC (Google Remote Procedure Call) as a modern universal standard.
