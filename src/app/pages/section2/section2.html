Lesson 1

Client-Server Architecture

The revolution that started it all — separating server and client into different locations where different pieces of code can live and execute remotely.

The Problem It Solves

  • Machines are expensive
  • Applications are complex
  • Running everything on a single mainframe is inefficient

Solution: Break the application into multiple components that can call each other.

Benefits

Centralized Servers

Servers have dedicated resources; clients use commodity hardware

Better Scaling

Many lightweight clients can call a single powerful server

Faster Startup

Clients don't carry heavy application logic; binaries are smaller

Fewer Dependencies

Server handles dependencies (DB drivers, libs); client just makes calls

Key insight: Microservices inherited this same model — breaking a monolith into smaller services that call each other.

Remote Procedure Call (RPC)

Born from client-server architecture in the 1960s-70s. The idea of making a call to a remote machine. Initially no standards — just send data across the wire. Standards evolved over time, leading to gRPC (Google Remote Procedure Call) as a modern universal standard.

OSI Model

A standard communication model with seven layers, each describing a specific networking component.

Why We Need a Communication Model

01

Agnostic Applications

Without a standard, apps must know the underlying network medium

02

Medium Independence

Same app works on Wi-Fi, Ethernet, LTE, fiber, satellite

03

Equipment Management

Devices communicate regardless of manufacturer

04

Layer Independence

Each layer can be improved separately without affecting others

Key insight: We take this for granted. A Node.js app sends requests without caring if it goes through Wi-Fi, fiber, or satellite.

Transparent vs Application Proxies

Transparent Proxy (L3-4)

  • Sees IP addresses and ports (always unencrypted)
  • Cannot see encrypted application data
  • ISPs use these to block websites by IP
  • Governments use these for censorship

Application Proxy (L7)

  • Must decrypt TLS to see content
  • CDNs (Cloudflare, Fastly) are L7 reverse proxies
  • Slower — must process all layers up and down
  • Creates two sessions: client↔proxy, proxy↔server

Key Takeaways

  • Know where your application sits in the model
  • Layer 4 and Layer 7 are where backend engineers spend most time
  • Lower layers = faster processing (less to parse)
  • Higher layers = more information but more overhead
  • Everything below Layer 5 (IP, ports) is public and unencrypted — anyone in the path can see it

Host-to-Host Communication

How messages are sent between hosts at the lowest level — Layer 2 and Layer 3.

MAC Address (Media Access Control)

Every network card has a unique MAC address: 00:1A:2B:3C:4D:5E (6 bytes, 48 bits). This is the lowest level of addressing.

The Broadcast Problem

On a shared network, when Host A sends to Host B:

Everyone receives the frame. Only the intended recipient processes it — others drop it. This is how public Wi-Fi sniffing worked.

Why MAC Addresses Aren't Enough

No Routability

MAC addresses are random — no location info

No Segmentation

Cannot divide into smaller networks

Global Scanning

Finding a MAC globally means scanning everything

Key insight: MAC addresses are globally unique but provide no location information — like having a unique name but no address.
