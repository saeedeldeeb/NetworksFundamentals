Section 04

UDP (User Datagram Protocol)

Understanding the simplest Layer 4 protocol — stateless, connectionless, and fast

Lesson 5

UDP Server with C

Going to the lowest level — building a UDP server in C to understand what high-level abstractions hide: buffers, file descriptors, byte order, and memory.

The Complete UDP Server

Key Components

1
Socket file descriptor

socket(AF_INET, SOCK_DGRAM, 0) returns an integer — a file descriptor. The OS uses it to identify your socket. Default limit: ~10,000 per process.

2
Address structures

In Node.js, addresses are simple strings like "127.0.0.1". In C, you fill a sockaddr_in struct with family, port, and address fields.

3
Buffer allocation

char buffer[1024] — you manually allocate the receive buffer. This is the max data you can receive at once (not including the 8-byte UDP header).

4
Byte order conversion

htons(port) converts from host byte order to network byte order. inet_pton() converts a string IP to binary. Node.js does this automatically.

5
Bind + Receive

bind() associates the socket with an IP:Port. recvfrom() blocks until a datagram arrives and fills remoteaddr with the sender's info.

Why does the program exit?

No loop! The program receives one datagram, prints it, and exits. C does exactly what you tell it — if you don't write a loop, it won't loop.

Node.js vs C Comparison

Task Node.js C
Buffer allocation Automatic Manual (arrays)
File descriptors Hidden Explicit integers
Event loop Built-in Write it yourself
Memory Garbage collected Manual management
Byte order Automatic htons / inet_pton
Address handling Simple strings Structs & conversions
Control Limited Complete
Key insight: High-level languages do all this for you in one or two lines. Understanding the C implementation helps you debug issues and understand what's happening when things go wrong — because under the hood, it's always sockets, buffers, and file descriptors.
