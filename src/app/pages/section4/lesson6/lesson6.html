Lesson 6

Capturing UDP Traffic with tcpdump

Seeing UDP in action by capturing real DNS traffic. Despite its name, tcpdump captures all protocols — including UDP, ICMP, and ARP.

The Capture Command

-n Show numbers only (no DNS resolution)
-vv Very verbose output
-i en0 Capture on interface en0
host 8.8.8.8 Filter: packets to/from Google DNS

Generate DNS Traffic

The Request Packet

IP
TTL 64 (max hops)
Protocol 17 = UDP
Total length 63 bytes
Source IP 192.168.1.144
Dest IP 8.8.8.8
UDP
Source port 58635 (ephemeral)
Dest port 53 (DNS)
Length 35 bytes
DNS
Query ID 12345
Type A (IP address lookup)

The Response Packet

Request vs Response

Field Request Response
Source IP 192.168.1.144 8.8.8.8
Source Port 58635 53
Dest IP 8.8.8.8 192.168.1.144
Dest Port 53 58635
TTL 64 120
Total Size 63 bytes 127 bytes

Notice how source and destination are flipped in the response — that's how the server knows where to reply. The same query ID (12345) links request to response.

Packet Size Breakdown

Request (63 bytes)
IP header 20 bytes
UDP header 8 bytes
DNS data 35 bytes
Small query
Response (127 bytes)
IP header 20 bytes
UDP header 8 bytes
DNS data 99 bytes (4 IP addresses)
Larger response — amplification potential
Key insight: UDP is just a thin wrapper that adds ports to IP. The real work happens in the application protocol (DNS in this case). Looking at a capture, UDP adds very little — source port, destination port, length, and checksum. This simplicity is what makes UDP fast and lightweight.
