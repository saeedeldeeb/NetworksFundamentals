Section 04

UDP (User Datagram Protocol)

Understanding the simplest Layer 4 protocol — stateless, connectionless, and fast

Lesson 1

What Is UDP?

The User Datagram Protocol — a simple, stateless Layer 4 protocol that sits on top of IP and introduces the concept of ports.

UDP at a Glance

Layer 4 (Transport)
Header 8 bytes
State Stateless
Connection None required

UDP sends datagrams — self-contained packets of data that don't require a connection or acknowledgment. It's the "send and forget" protocol.

UDP vs TCP

Feature UDP TCP
Connection required No Yes
State stored No (stateless) Yes (stateful)
Delivery guaranteed No Yes
Order guaranteed No Yes
Header size 8 bytes 20+ bytes
Speed Faster Slower

What Stateless Really Means

TCP (Stateful)
Setup 3-way handshake before data
Memory Stores window sizes, sequences, buffers, timers
Tracking Maintains connection state throughout
More overhead, more guarantees
UDP (Stateless)
Setup None — send immediately
Memory Nothing stored on either side
Tracking Each datagram is independent
Minimal overhead, maximum speed

The Power of Ports

Before UDP (at the IP layer), you could only address a host — an entire machine. UDP introduces ports, letting you address specific processes within a host.

UDP Use Cases

▶
Video Streaming
Dropped frames are acceptable. Retransmitting old frames would cause buffering — better to skip a frame than delay the stream.

VPN: Why Not TCP?

If a VPN uses TCP and your application also uses TCP, you get a dangerous problem called TCP Meltdown:

Layer 2 Wrapped in VPN's TCP

Double retransmission = TCP Meltdown
VPN over UDP (Good)
Layer 1 Your app's TCP packet
Layer 2 Wrapped in VPN's UDP
Loss Only your app's TCP handles retransmission

Multiplexing & Demultiplexing

Multiplexing combines multiple app streams into one network connection. Demultiplexing sorts incoming packets to the correct application by port.

The Four-Tuple

Every UDP datagram is uniquely identified by four values. Together, they tell the network exactly who is talking to whom:

+
Source Port Sender's app
+

The source port is critical — without it, the server wouldn't know which application on the client to send the response back to.

Lesson 2

UDP Datagram Structure

The anatomy of the UDP datagram — understanding the 8-byte header that makes UDP the simplest transport protocol.

How UDP Fits Inside IP

A UDP datagram slides into an IP packet as its data portion. The IP header's protocol field is set to 17, telling the receiver "this data is UDP." The IP packet doesn't care what's inside — it just sees bytes.

The UDP Header

Only 4 fields, each 16 bits (2 bytes). Two rows of 32 bits each give us the 8-byte header. Compare that to TCP's 20–60 bytes — UDP keeps it minimal.

Source & Destination Ports

Both are 16 bits, giving a range of 0–65,535. Ports identify which application on a host should receive the datagram.

Port Service Purpose
53 DNS Domain name resolution
67/68 DHCP Automatic IP assignment
69 TFTP Simple file transfer
123 NTP Time synchronization
161 SNMP Network monitoring
500 IKE VPN key exchange

Port Exhaustion

If you fix the destination IP, destination port, and source IP, then only the source port varies — limiting you to 65,535 concurrent connections to the same server.

Fixed
Dst IP: 8.8.8.8 Dst Port: 53 Src IP: 10.0.0.1
Variable
Src Port: 1024 ... 65535
Max ~65,535 connections

Length Field

Specifies the total size of the UDP datagram (header + data). At 16 bits, the maximum value is 65,535 bytes. Since the header is always 8 bytes, the minimum value is 8 (header only, no data).

Minimum
Header 8 bytes
Data 0 bytes
Length = 8
Maximum
Header 8 bytes
Data 65,527 bytes
Length = 65,535

Checksum Field

The checksum detects data corruption during transmission. Electrical interference, voltage attenuation, or even cosmic rays can flip bits in transit.

2

IPv4 vs IPv6: In IPv4, the UDP checksum is optional. In IPv6, it's mandatory — because IPv6 removed the IP-level checksum, so the transport layer must ensure integrity.

UDP vs TCP Header

Field UDP TCP
Source Port 16 bits 16 bits
Destination Port 16 bits 16 bits

Key insight: UDP's 8-byte header carries only what's essential — ports for addressing, length for sizing, and a checksum for integrity. No sequence numbers, no acknowledgments, no flow control. This simplicity means more bandwidth for your actual data and less processing at each hop.
Lesson 3

UDP Pros & Cons

Everything has tradeoffs. Understanding UDP's strengths and weaknesses helps you choose the right protocol for the job.

Pros of UDP

⚡
Simple Protocol
Does exactly what developers need — nothing more. No unnecessary overhead. Perfect when you want control over the communication logic.
📦
Small Header (8 bytes)
More bandwidth available for actual data. Less processing at each network hop. Faster packet handling.
📈
Stateless & Scalable
No memory consumed for connection tracking. Server can handle many more concurrent clients. Process and forget.
🚀
Low Latency
No handshake, no acknowledgments, no retransmission delays. First packet = your actual data.

Latency Comparison

TCP (4 packets before data)
Pkt 1 SYN
Pkt 2 SYN-ACK
Pkt 3 ACK
Pkt 4 DATA (finally!)
1.5 round trips before data
UDP (1 packet)
Pkt 1 DATA (immediately!)
Zero round trips — instant

Stateful vs Stateless

TCP: Each Connection Stores
Memory Window sizes, sequences, buffers, timers
Scale Memory grows with each connection
1000s of connections = significant RAM
UDP: Nothing Stored
Memory Process and forget — zero state
Scale Theoretically infinite clients
Scales with almost no overhead

Cons of UDP

1
No Acknowledgment Sender doesn't know if data arrived. Can't verify the server received or processed anything. Bad for databases, financial transactions, or any critical data.
2
No Guaranteed Delivery Packets may be lost or arrive out of order. No automatic retransmission. The application must handle losses itself.
3
Connectionless (Security Risk) No authentication between parties. Anyone can send to anyone. Source addresses are easily spoofed — enabling reflection attacks.
4
No Flow Control Sender has no idea how much the receiver can handle. Can overwhelm the receiver's buffer. Packets dropped silently with no feedback.
5
No Congestion Control No awareness of network conditions. Can contribute to congestion. Routers may drop packets silently with no backoff mechanism.
6
No Packet Ordering Packets may arrive out of sequence. The application must reorder them if order matters — or accept out-of-order data.

UDP-Based Attacks

UDP's connectionless, stateless nature makes it vulnerable to abuse. Two common attacks exploit these weaknesses:

DNS Amplification Attack

The attacker sends small DNS queries to many DNS servers but forges the source IP to be the victim's address. DNS servers send their large responses to the victim, flooding it with unsolicited traffic. A 40-byte query can produce a 3,000-byte response — a 75x amplification.

UDP Flood Attack

The attacker sends massive amounts of UDP datagrams to a server. Since UDP is stateless, the server must process every single packet — it can't filter by connection state like TCP can. This consumes CPU and memory until the server is overwhelmed.

Some developers choose UDP and build only the features they need. This gives maximum control — pick TCP-like features without the full cost.

Key insight: UDP gives you a blank canvas. TCP gives you a complete painting. Choose based on whether you need the full picture or want to paint your own.

UDP Server with Node.js

Building a UDP server using Node.js — a high-level approach that abstracts away buffers, file descriptors, and memory management.

The Complete UDP Server

Node.js provides the dgram module for UDP socket operations. The entire server fits in about 10 lines:

Step by Step

1
Import the dgram module

Node.js provides dgram for UDP socket operations. It handles all the low-level system calls behind the scenes.

2
Create a socket

createSocket('udp4') creates an IPv4 UDP socket. Use 'udp6' for IPv6. At this point the socket exists but has no binding — it doesn't know where to listen yet.

3
Bind to address and port

socket.bind(5500, '127.0.0.1') tells the socket to listen on port 5500 at localhost only.

4
Handle incoming messages

The 'message' event fires for each datagram received. The callback provides the data and sender info (IP, port, family, size).

Local-only, no external access

Key insight: Node.js abstracts away buffers, file descriptors, loops, and memory management. The same functionality that takes ~30 lines in C takes ~10 in Node.js. But understanding the low-level implementation helps you know what's happening under the hood.
Lesson 5

UDP Server with C

Going to the lowest level — building a UDP server in C to understand what high-level abstractions hide: buffers, file descriptors, byte order, and memory.

The Complete UDP Server

Key Components

1
Socket file descriptor

socket(AF_INET, SOCK_DGRAM, 0) returns an integer — a file descriptor. The OS uses it to identify your socket. Default limit: ~10,000 per process.

2
Address structures

In Node.js, addresses are simple strings like "127.0.0.1". In C, you fill a sockaddr_in struct with family, port, and address fields.

3
Buffer allocation

char buffer[1024] — you manually allocate the receive buffer. This is the max data you can receive at once (not including the 8-byte UDP header).

4
Byte order conversion

htons(port) converts from host byte order to network byte order. inet_pton() converts a string IP to binary. Node.js does this automatically.

5
Bind + Receive

bind() associates the socket with an IP:Port. recvfrom() blocks until a datagram arrives and fills remoteaddr with the sender's info.

Why does the program exit?

No loop! The program receives one datagram, prints it, and exits. C does exactly what you tell it — if you don't write a loop, it won't loop.

Node.js vs C Comparison

Task Node.js C
Buffer allocation Automatic Manual (arrays)
File descriptors Hidden Explicit integers
Event loop Built-in Write it yourself
Memory Garbage collected Manual management
Byte order Automatic htons / inet_pton
Address handling Simple strings Structs & conversions
Control Limited Complete
Key insight: High-level languages do all this for you in one or two lines. Understanding the C implementation helps you debug issues and understand what's happening when things go wrong — because under the hood, it's always sockets, buffers, and file descriptors.
