TCP (Transmission Control Protocol)

The backbone of reliable internet communication — stateful, connection-oriented, and controlled

Lesson 1

What Is TCP?

The Transmission Control Protocol — one of the most popular protocols on the internet. Unlike UDP's "fire and forget" approach, TCP controls the transmission.

TCP vs UDP: The Core Difference

UDP is a firehose — it just sends data without control, which can overwhelm receivers and networks. TCP controls the flow: flow control, congestion control, ordering, and retransmission.

The Concept of a Connection

A TCP connection is an agreement between client and server, established through a handshake. It creates state (a session) on both sides, stored in memory as a file descriptor.

🔒
Security Can't just send data without establishing identity — no spoofing
📈

⚙
Control Can block or reject unwanted connections before any data flows

The Three-Way Handshake

Before any data can flow, TCP establishes a connection using three messages. SYN = Synchronize — both sides exchange sequence numbers so they can track data order.

1
SYN

Client says: "I want to connect. My sequence starts at 1000"

2
SYN-ACK

Server says: "OK. My sequence starts at 5000. I acknowledge your 1000"

3
ACK

Client says: "I acknowledge your 5000. Let's go!"

After the handshake, both sides have a file descriptor (socket) with each other's sequence numbers and connection metadata.

Sending Data & Acknowledgments

Once connected, the client sends segments with sequence numbers. The server acknowledges receipt — ACKing sequence N means "I received everything up to N."

You don't have to wait for each ACK before sending the next segment — you can send multiple at once. But there's a limit: flow control (how much the receiver can handle) and congestion control (how much the network can handle).

What If Segments Arrive Out of Order?

TCP segments travel inside IP packets, and IP has no guaranteed path — different routers may choose different routes. The receiver uses sequence numbers to reconstruct the original order before delivering to the application.

1 2 3
→

→

Retransmission: TCP's Reliability

When segments are lost (router congestion, network failure, timeouts), TCP automatically retransmits them. The sender notices missing ACKs and resends the data.

Closing a Connection: Four-Way Handshake

Each side must independently announce they're done (FIN) and acknowledge the other's FIN (ACK). The side that initiates the close enters TIME_WAIT — waiting ~4 minutes for any delayed segments.

Why TIME_WAIT? To catch any old, delayed segments that might arrive late and prevent confusion if a new connection reuses the same four-tuple. It's not a bug — it's a safety mechanism.

Bidirectional Communication

TCP itself is bidirectional — either side can send data at any time, like WebSockets. HTTP adds a request-response pattern on top, but TCP doesn't require it.

TCP A ↔ B Either side can send anytime
HTTP A → B → A Request-response pattern on top

Chat / Messaging

🖥
Remote Shell (SSH)

🗃
Databases
Can't have SELECT become TRUNCATE. Query integrity is critical.
🌐
Web (HTTP/1.1, HTTP/2)
Reliable page loading. Every resource must arrive complete and in order.
📧
Email
Messages must arrive complete. Partial emails are useless.
📁
File Transfer
Files can't have missing pieces. Every byte must arrive correctly.

Connection Limits

Each connection stores state in memory: sequence numbers, window sizes, buffers, timers. More connections = more memory. WhatsApp servers handle up to 3 million TCP connections per server.

"But ports only go to 65,535?" That limit applies to connections TO a single destination. Connections FROM different source IPs are unlimited. Theoretical max: 4 billion IPs × 65,535 ports. Practical limits: RAM, CPU, and OS file descriptor limits.

Connection Lookup Cost

Every incoming segment requires the OS to:

1 Hash the four-tuple
2 Look up connection table
3 If found → process
4 If not found → drop/reject

At millions of connections, this consumes significant CPU. Unlike UDP, where any packet is processed, TCP rejects packets that don't belong to a valid connection.

Key insight: TCP's reliability comes at a cost — header overhead, memory for state, latency for handshakes, CPU for connection management. It controls the transmission, unlike UDP's firehose approach. Choose TCP when reliability matters more than raw speed.
