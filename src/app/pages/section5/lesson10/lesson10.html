Lesson 10

TCP Server with JavaScript (Node.js)

Building a practical TCP server with Node.js reveals how high-level APIs abstract away kernel complexity — your callback fires only after the three-way handshake is complete.

The Code

The Socket Callback

When net.createServer() fires your callback, the full three-way handshake is already done. You never see SYN or SYN-ACK — the kernel handles them invisibly.

1
SYN received

Client initiates — kernel places connection in SYN Queue, no app code runs

2
SYN-ACK sent

Kernel responds automatically — your application never sees this packet

3
ACK received

Handshake complete — connection moves from SYN Queue to Accept Queue

4
Your callback fires

Node.js calls accept() and delivers the established connection to you

Socket Properties

Property Example Description
socket.remoteAddress 127.0.0.1 Client's IP address
socket.remotePort 64409 Client's ephemeral port (assigned by their OS)

Always Specify the Listen Address

The Danger of Omitting the Address

server.listen(8800) without an IP listens on 0.0.0.0 — all interfaces, including any public-facing IP.

This is how MongoDB and Elasticsearch breaches happened — databases deployed to cloud VMs, listening on 0.0.0.0, exposed to anyone scanning the internet.

Always write server.listen(8800, '127.0.0.1') for local-only services.

Testing with Netcat

Netcat without -u defaults to TCP. It performs the handshake and enters interactive mode — every line you type is sent as a TCP segment.

Multiple Connections

Open a second terminal and connect again. Each connection gets its own socket object with a unique ephemeral port. Both are handled concurrently.

Each connection has its own file descriptor, its own Send/Receive queues in the kernel, and its own isolated closure in Node.js — the socket variable in each callback refers to a completely separate connection.

The Cost of Small Messages

Hitting Enter in netcat sends a single newline character — just 1 byte. Look at what the network actually transmits:

Your data 1 byte
TCP Header 20 bytes
IP Header 20 bytes
97% overhead for a single character. This is why Nagle's algorithm buffers small writes, why protocols batch messages, and why connection reuse matters.

TCP vs UDP at a Glance

Aspect TCP (net module) UDP (dgram module)
Create net.createServer() dgram.createSocket()
Connect with nc host port nc -u host port
Handshake Required (callback fires after) None (immediate)
Per-client object Yes — one socket per connection No — one socket, use rinfo
Client identity Automatic (remoteAddress/Port) Manual (rinfo in each callback)

What's Really Happening Under the Hood

Watch how a client's connection request travels through the abstraction layers to reach your callback. The kernel handles the handshake entirely — your code never sees a SYN packet.

Key insight: The socket callback represents a fully established connection. By the time your code runs, the kernel has completed the three-way handshake and placed the connection in the Accept Queue. Node.js wraps this in a clean event-driven API — but understanding the layers beneath helps you diagnose connection failures, slow accept rates, and backlog exhaustion under load.
