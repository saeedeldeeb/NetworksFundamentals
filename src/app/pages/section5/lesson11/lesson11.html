TCP (Transmission Control Protocol)

The backbone of reliable internet communication — stateful, connection-oriented, and controlled

Lesson 11

TCP Server with C

Building a TCP server in C exposes every syscall that high-level languages hide. Seeing the raw steps — socket(), bind(), listen(), accept() — makes the kernel's role concrete.

The Complete Server

TCP vs UDP: Two Extra Steps

TCP requires listen() and accept() — UDP has neither. These are the calls that create the kernel queues and block waiting for clients.

Step UDP TCP
Create socket SOCK_DGRAM SOCK_STREAM
Bind Yes Yes
Listen No Yes — creates SYN/Accept Queues
Accept No Yes — blocks, returns new fd
Per-client fd No — one socket for all Yes — new fd per connection

The Syscall Sequence

Watch each syscall step by step — what it does, what fd it creates, and what the kernel sets up underneath.

Step-by-Step Breakdown

1
socket(AF_INET, SOCK_STREAM, 0)

Creates a TCP socket file descriptor (socket_fd = 3). SOCK_STREAM = TCP. Use SOCK_DGRAM for UDP. Nothing is bound yet — just a socket that exists.

2
bind(socket_fd, &server_addr, sizeof(server_addr))

Tells the kernel which IP and port this socket owns. INADDR_ANY means all interfaces (0.0.0.0) — use a specific IP in production. htons(PORT) converts to network byte order.

3
listen(socket_fd, 5)

Enables the TCP handshake machinery. Creates the SYN Queue and Accept Queue. The second argument — the backlog — caps how many connections can wait in the Accept Queue. Too small = connections dropped under load.

4
new_socket = accept(socket_fd, &client_addr, &addr_len)

Blocks until a connection is ready in the Accept Queue. Returns a brand-new file descriptor (new_socket = 4) for this specific connection. client_addr holds the client's IP and port. socket_fd continues listening.

5
send(new_socket, buffer, strlen(buffer), 0)

Writes data to the connection socket — always new_socket, not socket_fd. You manage the buffer manually: strcpy() the data in, pass the length explicitly.

6
close(new_socket) + close(socket_fd)

close(new_socket) triggers the four-way FIN handshake with the client. close(socket_fd) stops the listening socket. Forgetting either call leaks file descriptors — a common C server bug.

The Backlog Problem

If your application is slow to call accept(), the Accept Queue fills up and new connections are silently dropped.

Event Accept Queue Result
Client 1 connects [1] Queued, waiting
Clients 2–5 connect [1, 2, 3, 4, 5] Queue full (backlog=5)
Client 6 sends SYN [full] No SYN-ACK — connection times out
Solutions: Increase backlog size, call accept() in a tight loop, use a thread pool (main thread accepts, workers handle connections), or switch to async I/O (epoll, kqueue).

Handling Multiple Connections

This server accepts one connection and exits. Real servers loop forever, spawning a thread or process per connection.

Node.js handles this automatically — each callback in createServer() gets its own socket closure and is driven by libuv's event loop without blocking.

Key insight: High-level languages hide enormous complexity. The net.createServer() callback that "just works" is backed by decades of systems programming: efficient accept loops, memory management, threading models, and async I/O. Understanding the C version makes you appreciate what's happening underneath — and helps you diagnose performance issues when the abstraction breaks down.
