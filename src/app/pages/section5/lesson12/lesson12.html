TCP (Transmission Control Protocol)

The backbone of reliable internet communication — stateful, connection-oriented, and controlled

Lesson 12

Capturing TCP Segments with tcpdump

Watching TCP on the wire makes every concept concrete. A single HTTP request to example.com produces 10 packets — three to say hello, four to exchange data, and three to say goodbye.

Setup

tcpdump captures raw packets on a network interface. Run it as root while a browser fetches a page over plain HTTP (port 80) so you can see the TCP layer clearly.

The Full Exchange

Scroll through the complete lifecycle — handshake, request, response, and teardown — as packets appear in the order they arrive on the wire.

Breaking Down Each Phase

Phase 1 — Handshake (3 packets, 176 B, zero user data)

The SYN packet carries the client's initial options:

Field Value Meaning
Flags [S] SYN Starting handshake
flags [DF] Don't Fragment Path MTU Discovery
mss 1460 Max Segment Size MTU(1500) − IP(20) − TCP(20)
wscale 6 Window Scale Multiply window by 2^6 = 64
length 0 No data Just handshake overhead

The server replies with its own sequence number and window scale:

Notice: Server uses wscale 9 (×512) vs client's wscale 6 (×64) — the server is ready to receive more data in flight, signaling higher throughput capacity. Also, the server sets no [DF] flag — different OS defaults.

Phase 2 — Data Transfer (4 packets)

Field Value Meaning
Flags [P.] PUSH + ACK Send immediately, don't buffer
seq 1:361 Sequence range 360 bytes of request data
length 412 IP packet total Headers (52 B) + data (360 B)

Phase 3 — Teardown (3 packets, 156 B)

Safari's idle connection timeout is 30 seconds — discoverable just by watching the wire. The client sends FIN, the server echoes FIN back, the client confirms with the final ACK.

TCP Flag Reference

Symbol Flag Meaning
[S] SYN Synchronize — start connection
[.] ACK Acknowledgment
[S.] SYN + ACK Combined SYN and ACK
[P.] PUSH + ACK Send data immediately
[F.] FIN + ACK Finish — close connection
[R] RST Reset — abort connection immediately

Key Observations

1
Handshake is expensive

176 bytes across 3 packets — zero user data. This is why TCP Fast Open and QUIC were invented: they send data inside the handshake itself.

2
Empty ACKs add up

Every 52-byte ACK carries zero data. In a busy stream, pure ACKs consume bandwidth and CPU. TCP's delayed ACK (200ms wait) batches acknowledgments to reduce this.

3
NAT translation is invisible here

tcpdump captures before NAT, so you see your private IP (192.168.x.x). The server never sees it — your router rewrites the source IP to its public address before forwarding.

4
Application behavior is in the packets

Safari's 30-second idle timeout is directly visible — the FIN arrives exactly 30s after the last data packet. Packet captures let you reverse-engineer application behavior without any source code.

The Overhead in Numbers

Phase Packets Total / User Data
Handshake 3 176 B / 0 B

Total 10 1922 B / 1382 B (72%)

Saving and Replaying Captures

Key insight: Packet captures are a debugging superpower. You can see handshake timing, window sizes, retransmissions, idle timeouts, and OS-level differences — all without touching application code. Any production network issue that can't be diagnosed from logs alone can usually be understood by watching the wire. This is why network engineers reach for tcpdump and Wireshark first.
