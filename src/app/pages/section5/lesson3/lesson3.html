TCP (Transmission Control Protocol)

The backbone of reliable internet communication — stateful, connection-oriented, and controlled

Lesson 3

Flow Control

How much data can the receiver handle? Flow control prevents the sender from overwhelming the receiver's buffer.

Flow Control vs Congestion Control

These are often confused but solve different problems:

Flow Control
How much can the receiver handle?
Protects the receiver
Congestion Control
How much can the network handle?
Protects the routers

The Problem: Naive Approach

Send one segment, wait for ACK, send next, wait for ACK... You pay the round-trip time (RTT) for every single segment. Extremely slow.

The better approach: send multiple segments at once, get one ACK for all of them. But how many can you send? That depends on the receiver's buffer.

The Receiver Buffer

When segments arrive, they go into a receiver buffer in memory. The application reads from this buffer at its own pace. If you send too much data:

1 Buffer fills up
2 New segments are dropped
3 Wasted bandwidth, need retransmission

The Solution: Receiver Window (rwnd)

The receiver tells the sender: "This is how much data I can receive right now." This value is carried in the Window Size field of every TCP ACK.

Receiver Window (rwnd) Flow Control Window Window Size (TCP header)

Flow Control in Action

Watch how the receiver dynamically adjusts the window size based on buffer capacity. When the buffer is full, the sender must stop until the receiver catches up.

Dynamic Window Sizes

The receiver window changes with every ACK — it's not static:

ACK 1
5000 B "Lots of room"
ACK 2
3000 B "Filling up"
ACK 3
1000 B "Almost full!"
ACK 4
0 B "STOP!"
ACK 5
4000 B "App read data, room again"

The Sliding Window

As ACKs arrive, the sender "slides" its window forward. Acknowledged data is discarded, and new data enters the window.

Window Scaling

The Window Size field is only 16 bits — max 65,535 bytes. That's nothing for modern networks. The solution: Window Scaling (RFC 1323).

Actual Window = Window Size × 2scale factor
Window Size in header 65,535
Scale factor 14
Actual Window 65,535 × 16,384 ≈ 1 GB
Important: Window scaling is negotiated only during the handshake. If Wireshark misses the handshake, it can't calculate the actual window size — you'll see "window size" vs "calculated window size" in captures.

Zero Window

When the receiver sends Window = 0, the sender must stop sending data. It periodically sends tiny window probes to check if the receiver has room again.

Without Flow Control (UDP)

UDP has no flow control built in. If you need it, you must implement it yourself: application-level window negotiation, custom acknowledgments, your own buffering logic. Or just assume your servers are "beefy enough" — risky with many clients.

Key insight: Flow control protects the receiver from being overwhelmed. The receiver advertises its buffer capacity, and the sender respects that limit. Without this, fast senders would constantly overflow slow receivers' buffers, causing packet loss and retransmissions.
