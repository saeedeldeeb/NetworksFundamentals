TCP (Transmission Control Protocol)

The backbone of reliable internet communication — stateful, connection-oriented, and controlled

Lesson 7

TCP Connection States

TCP is stateful — connections go through a state machine with defined transitions. Getting states wrong causes bugs, resource leaks, and security issues.

Why States Matter

Both client and server maintain state for every connection: window sizes, sequence numbers, acknowledgment numbers, and the connection state itself — open, closing, closed, waiting.

The Four-Way Close State Machine

Watch how states transition during a connection close. The active closer (whoever sends the first FIN) goes through more states and ends up in TIME_WAIT.

State Descriptions

ESTABLISHED Both Connection open, data flowing normally
FIN_WAIT_1 Active closer Sent FIN, waiting for ACK
CLOSE_WAIT Passive closer Received FIN, sent ACK, preparing to close
FIN_WAIT_2 Active closer Got ACK for FIN, waiting for peer's FIN
LAST_ACK Passive closer Sent FIN, waiting for final ACK
TIME_WAIT Active closer Sent final ACK, waiting ~4 minutes for old packets to expire
CLOSED Both Connection fully terminated

TIME_WAIT: The Most Important State

After sending the final ACK, the active closer enters TIME_WAIT for 2 × MSL (Maximum Segment Lifetime) ≈ 4 minutes. This is the most discussed state in backend engineering.

Why TIME_WAIT Exists

The Two Generals' Problem: the client sends the final ACK, but there's no ACK for the ACK. Did the server receive it? If we close immediately:

1 Final ACK might be lost
2 Server retransmits FIN
3 Client already closed → sends RST
! Connection terminated uncleanly

The Security Reason

Without TIME_WAIT, a session resumption attack is possible:

Connection A uses port 45678, then closes
↓
Attacker quickly opens new connection, gets port 45678
↓
Old delayed packets from A arrive → attacker receives data meant for A

TIME_WAIT prevents this by ensuring old packets have time to expire before the port can be reused.

Who Gets TIME_WAIT?

Rule: Whoever sends the first FIN ends up in TIME_WAIT. If the server initiates the close, the server gets TIME_WAIT. If the client initiates, the client gets it.

The Backend Engineering Trick

Problem: Server has thousands of TIME_WAIT connections eating resources. Why? Because the server is initiating closes.

Bad for server
Server → FIN → Client
Server ends up in TIME_WAIT
Good for server
Server → "please close" → Client → FIN → Server
Client ends up in TIME_WAIT

Why this works: you have many clients but one server. Clients are distributed and can handle TIME_WAIT. Server resources are precious and centralized.

Viewing Connection States

$ netstat -an | grep tcp | sort by state | count
1234 ESTABLISHED
5678 TIME_WAIT
23 CLOSE_WAIT
5 FIN_WAIT_2

Common Issues

TIME_WAIT accumulation
Symptom Port exhaustion, can't open new connections
Cause Server closing many short-lived connections
Fix Have clients close instead, use connection pooling, tune tcp_tw_reuse
CLOSE_WAIT accumulation
Symptom Connections stuck in CLOSE_WAIT forever
Cause App received FIN but never called close()
Fix Application bug — always close sockets when done
Key insight: TIME_WAIT is not a bug — it's a feature that ensures clean connection termination and prevents security issues. But it can cause resource problems on busy servers. The trick is to design your protocol so clients initiate the close, pushing TIME_WAIT to the client side where it matters less.
