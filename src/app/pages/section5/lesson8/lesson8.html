Section 05

TCP (Transmission Control Protocol)

The backbone of reliable internet communication — stateful, connection-oriented, and controlled

Lesson 8

TCP Pros and Cons

Every protocol has trade-offs. TCP's pros are essentially UDP's cons, and TCP's cons explain why protocols like QUIC were invented.

TCP Pros

Guaranteed delivery Retransmits lost segments automatically
Connection required No data without handshake — prevents IP spoofing
Flow control Receiver controls how much the sender can transmit
Congestion control Prevents network collapse, fair bandwidth sharing
Ordered packets No application-level reordering needed
Harder to DoS Can't flood with spoofed packets — need valid connection
Security note: TCP prevents easy spoofing because you need the handshake. But TCP alone doesn't prove identity — you still need TLS certificates for that.

TCP Cons

Large header 20–60 bytes vs UDP's 8 bytes
Stateful Each connection consumes memory (especially TIME_WAIT)
Higher latency Slow start, congestion control, ACKs add delay
Complexity Flow control, congestion control, retransmission machinery
Head-of-line blocking One lost segment blocks ALL subsequent data

Head-of-Line Blocking: The Big Problem

TCP's most significant limitation — and the reason QUIC was invented. Multiple independent HTTP requests share one TCP connection. If one segment is lost, all requests behind it are blocked.

Requests 2, 3, and 4 have nothing to do with request 1. They're independent. But TCP only sees a stream of bytes that must be ordered — it doesn't know about application-level boundaries.

Why QUIC Was Invented

QUIC (HTTP/3) solves head-of-line blocking by running over UDP with multiple independent streams:

TCP Connection
Seg 1: /request1 — LOST
Seg 2: /request2 — BLOCKED
Seg 3: /request3 — BLOCKED
Seg 4: /request4 — BLOCKED
QUIC Connection
Stream 1: /request1 — LOST
Stream 2: /request2 — Delivered
Stream 3: /request3 — Delivered
Stream 4: /request4 — Delivered

Bad Patterns to Avoid

Single TCP for multiple DB queries

Query 3 finishes first, but its response waits for Query 1 and 2. Also: how do you know which response belongs to which query without request IDs?

Fix: connection pooling or multiplexing protocol with IDs
TCP-over-TCP (VPNs)

Two TCP stacks fight each other — both detect loss, both retransmit, both reduce congestion window. System spirals into terrible performance.

Fix: VPNs should use UDP (like WireGuard)

When TCP's Overhead Hurts

Scenario Why TCP hurts
High-frequency trading Every microsecond counts; UDP often used
Real-time gaming Old data is useless; fresh packets over retransmit
Video streaming Late frames cause stutter; skip rather than wait
Short-lived connections Handshake overhead dominates
Multiplexed protocols Head-of-line blocking kills performance

The Trade-off Summary

Aspect TCP Advantage TCP Disadvantage
Reliability Guaranteed delivery Retransmission latency
Ordering Automatic Head-of-line blocking
State Connection security Memory consumption
Control Flow + congestion Slow start delay
Header Rich features 20–60 byte overhead
Key insight: TCP tries to do a lot at Layer 4. For many modern use cases (HTTP/2+, gaming, streaming), this "help" becomes a hindrance. QUIC moves these features to Layer 7 where the application can make smarter decisions about what actually needs ordering and reliability.
