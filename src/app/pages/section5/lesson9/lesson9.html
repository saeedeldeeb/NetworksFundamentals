TCP (Transmission Control Protocol)

The backbone of reliable internet communication — stateful, connection-oriented, and controlled

Lesson 9

Sockets, Connections, and Kernel Queues

Understanding the kernel data structures behind TCP connections — sockets, queues, and how the kernel manages the handshake before your application ever sees it.

What Is a Socket?

A socket is a file descriptor — a number your OS uses to represent a network endpoint. There are two types:

Listening Socket
Represents the IP:Port you're accepting connections on
SYN Queue Accept Queue
Connection Socket
Represents one established connection (has src + dst)
Send Queue Recv Queue

Listening on IP and Port

You must specify both an IP and a port. A machine can have multiple network interfaces, each with its own IP. You choose which one to listen on.

The Danger of 0.0.0.0

0.0.0.0 means listen on ALL interfaces. If one interface has a public IP, your app is exposed to the internet.

This is how MongoDB and Elasticsearch breaches happened — databases listening on 0.0.0.0, deployed to cloud, attackers scan all IPv4 ranges.

Always specify the exact IP you intend to listen on.

The Two Queues: SYN Queue & Accept Queue

Every listening socket has two queues. The backlog parameter controls their size limit — if they fill up, new connections are dropped.

Connection Establishment Flow

Watch how a connection moves through the kernel: the client's SYN enters the SYN Queue, the handshake completes in the kernel, and the finished connection waits in the Accept Queue until your app calls accept().

The SYN Flood Attack

An attacker sends many SYNs but never completes the handshake (never sends the final ACK). The SYN Queue fills up and legitimate clients can't connect.

What Happens After accept()

When your app calls accept(), the connection is removed from the Accept Queue and a new file descriptor is created with its own Send and Receive queues.

The Complete Picture

A backend process holds file descriptors for one listening socket and multiple connections. The kernel manages all the queues underneath.

Scaling Connection Acceptance

A single process calling accept() becomes a bottleneck under load. Two approaches exist to scale — Fork and SO_REUSEPORT — and they work very differently.

Fork: Shared Socket

When a process forks, the child inherits the parent's file descriptors — including the listening socket. Both processes call accept() on the same socket with the same queues. The problem: all processes compete for one Accept Queue (thundering herd).

SO_REUSEPORT: Separate Sockets

With SO_REUSEPORT, each process creates its own socket on the same IP:Port. Each gets its own SYN and Accept queues. The kernel load-balances incoming connections across the sockets — no contention. Used by Nginx, Envoy, HAProxy.

Security: Without protection, any process could use SO_REUSEPORT to hijack a port. Solution: the first process sets a key — others must know the key to share.

Problems with Connection Acceptance

Problem Cause Effect
Backend too slow Not calling accept() fast enough Accept Queue fills up
Clients don't ACK Malicious or broken clients SYN Queue fills up
Small backlog Default settings Queues fill quickly
Blocking accept accept() blocks if empty Thread stuck waiting
Why async I/O matters: accept() is a blocking call. If no connections are ready, your process waits. This is why epoll, kqueue, and io_uring exist — they let you handle connections without blocking.
Key insight: Your application code lives far above where connections are actually managed. The kernel handles the handshake, maintains queues, and creates data structures. Understanding these internals helps you diagnose why connections fail, why servers slow down under load, and how to scale connection acceptance.
